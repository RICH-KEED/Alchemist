# Store & Docs Publisher

A **publication super-skill** that bundles the scattered doc + store skills (18, 22, 64, 65, 59) and fills the critical gap they all share: **auto-generating Play Store listing images at exact dimensions with device frames.**

When a project hits stages 22→24, it needs eight deliverables touching five different skills. This skill does them all in one pass, in the right order, with a single checklist and a single output bundle under `publish/`.

---

## What it generates

| Deliverable | Source | Output path |
|---|---|---|
| README | Codebase + PRD | `README.md` (generated or updated) |
| Architecture docs | Feature tree + CONVENTIONS | `docs/ARCHITECTURE.md` |
| ADRs | Decision Ledger (#63) | `docs/adr/` (new + index) |
| dartdoc | Public API | `doc/api/` (generated via `dart doc`) |
| Play short description (80 chars) | PRD + product positioning | `publish/play_short_description.txt` |
| Play full description (4000 chars) | README + feature list | `publish/play_full_description.txt` |
| What's New (500 chars) | Changelog (#65) | `publish/play_whats_new.txt` |
| **Feature graphic** (1024×500 px) | App name + brand token + screenshot/gradient | `publish/play_feature_graphic.png` |
| **Phone screenshots** (2–8, min 320px wide) | App screens (golden/preview or adb capture) | `publish/play_phone_1.png` … `play_phone_8.png` |
| **Tablet screenshots** (7" and 10", 2–8 each) | Same screens on tablet layouts | `publish/play_tablet_7in_1.png` … |
| Privacy policy | SDK/permission scan (#59) | `publish/privacy_policy.md` |
| Release checklist | Stages 22/24 | `publish/release_readiness.md` |

## The image generation pipeline (the missing piece)

Play Store listing images are the hardest part — they need exact dimensions, device frames, and a consistent aesthetic. This skill generates them via three methods (auto-select based on tool availability):

### Method A: Pencil MCP (preferred when connected)

1. `mcp__pencil__get_editor_state` → open a `.pen` canvas sized to each required Play dimension.
2. `mcp__pencil__batch_design` → generate a feature graphic from the app's brand token (skill 04: `ColorScheme` primary, `AppTokens`, app name in display type, optional screenshot inset).
3. For screenshots: capture individual screens via `mcp__pencil__get_screenshot`/`export_nodes` from the skill-05 `.pen` mockup, then batch_design them into device-frame templates (see `templates/play_dimensions.md`).
4. `mcp__pencil__export_nodes` → export as PNG.

### Method B: widget gallery → golden + frame

1. Render the screen in the skill-05 widget gallery at a fixed Play Store screenshot size.
2. Capture via golden test (`matchesGoldenFile`) or `flutter screenshot --device-id`.
3. Composite into a device frame using the `templates/device_frames/` SVG templates (free/open-source device-frame SVGs borrowed from the Android marketing kit).
4. Resize to exact Play pixel dimensions via ImageMagick (`convert` CLI; note the dependency).

### Method C: adb screenshot + frame

1. `adb shell screencap -p /sdcard/screen.png` → `adb pull`.
2. Crop to device resolution, composite into the frame, resize to Play dimensions.
3. Repeat for light and dark themes.

### Feature graphic template

The feature graphic (1024×500 px "header" at top of Play listing) is auto-generated by composing:
- **Background:** `ColorScheme.primary` or a gradient from brand tokens
- **App name:** display type, `onPrimary` or white, centered
- **Optional tagline:** one-line value prop from short description
- **Optional device inset:** a framed phone screenshot at ~30% opacity, aligned right

This is authored as a Pencil `.pen` template or an ImageMagick `convert` composition script in `scripts/generate_feature_graphic.sh`.

## The execution flow

```
1.  Collect inputs     read PRD (#02) + STATE.md + codebase index (#26)
2.  Generate docs      delegate to #18 patterns, fill README/ARCH/ADRs
3.  Generate dartdoc   run `dart doc` + verify no broken links
4.  Generate copy      short desc (PRD one-liner), full desc (features+benefits),
                       whats-new (changelog diff via #65)
5.  Generate images    (see pipeline above) — feature graphic + screenshots
6.  Generate privacy   delegate to #59 scan + template
7.  Package bundle     write `publish/` folder, ready for Play Console / fastlane supply
8.  Run checklist      verify every file + dimension (see templates/)
```

Steps 2–6 can run in parallel once inputs are collected.

## Play Store image dimension reference

See [`templates/play_dimensions.md`](templates/play_dimensions.md) for the full spec. Quick-reference:

| Asset | Dimensions (px) | Max size | Format |
|---|---|---|---|
| Feature graphic | 1024×500 | 1 MB | PNG/JPG 24-bit |
| Phone screenshot | min 320px wide, 16:9–9:16 | 8 MB | PNG/JPG |
| Tablet 7" screenshot | min 320px wide | 8 MB | PNG/JPG |
| Tablet 10" screenshot | min 320px wide | 8 MB | PNG/JPG |
| TV banner (optional) | 1280×720 | — | PNG/JPG |
| Wear screenshot (optional) | min 320px wide, 1:1 | — | PNG/JPG |
| App icon | 512×512 | 1 MB | PNG 32-bit (with alpha) |

## Integration notes

- Feeds directly into **skill 22** (`fastlane supply` picks up `publish/` for automated uploads).
- Runs as the final pre-launch step before skill **24 Production Readiness** signs off.
- The feature graphic and screenshots are the most visible part of a Play listing — they directly drive conversion (Play experiments show 35%+ CVR lift from good listings).
- Screenshots MUST show real app UI, not mockups, per Play policy.
- All copy is localized if skill **50 i18n** is active (ARB keys instead of raw English).

## Quality bar

- Feature graphic renders at exact 1024×500 in both a light and dark variant.
- Screenshots use a consistent device frame across all images.
- README passes the GitHub readability test (scan at 250 px width).
- All store copy fits Play character limits exactly.
- dartdoc generates with zero warnings.
