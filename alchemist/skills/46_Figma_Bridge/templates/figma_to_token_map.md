## Token conflict resolution rules

When multiple Figma values map to the same token or a token value disagrees with
the design, apply these rules in order:

1. **Code is truth.** The Dart tokens in `lib/app/theme/app_tokens.dart` and
   `lib/app/theme/color_schemes.dart` are authoritative. If Figma disagrees,
   flag the discrepancy — do NOT silently override code from design.
2. **Designer intent is surfaced.** Discrepancies appear in the gap register
   with the Figma value and the current code value side-by-side, so the
   designer can decide.
3. **New tokens follow skill 04 procedure.** Any token not in the existing
   catalogue must be added through skill 04's defined process (update
   `AppTokens` or `ColorScheme`, add light + dark variants, run `flutter analyze`).
4. **Dark mode is not optional.** Every new color token must have a dark variant
   defined before the map is considered complete. Use
   `ColorScheme.fromSeed(brightness: Brightness.dark)` as the baseline for
   dark-variant derivation.
5. **Naming conventions win ties.** If a hex value matches two tokens equally
   well (within the same delta), prefer the one whose variable name matches
   the convention (`color/surface/primary` → `colorScheme.primary`).

## How to add new tokens to AppTokens from Figma

When a Figma variable or style has no matching token:

1. **Catalog the gap** in the unmatched register above with the exact Figma
   value and suggested token name.
2. **Open skill 04** — the token addition must follow its procedure:
   - For a new semantic color: add light + dark hex values to
     `SemanticPalette` in `color_schemes.dart`, then wire through
     `AppSemanticColors.light` and `.dark`.
   - For a new spacing/ radius/ elevation value: add it to the matching
     `AppSpacing` / `AppRadius` / `AppElevation` class.
   - Run `flutter analyze` to confirm the token compiles.
3. **Update this map** — add the resolved row with confidence `exact`.
4. **Propagate downstream** — if widget stubs were already generated,
   regenerate them so they reference the new token.

Never add tokens directly from this skill — #46 identifies gaps; #04 fills them.
