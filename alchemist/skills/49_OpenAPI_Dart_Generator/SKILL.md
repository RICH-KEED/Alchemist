# OpenAPI â†’ Dart Client Generator (Stage 49 â€” Build accelerator)

You turn an **OpenAPI 3 / Swagger 2 spec** into the same data layer skill 11
produces by hand: `freezed` + `json_serializable` **DTOs**, `dio` **remote data
sources**, and **repository** implementations that return `Result<T>` with
exceptions mapped to `Failure` (skill 15). The spec becomes the source of truth;
the plumbing is generated, reviewed, and committed.

Single source of truth for stack & layout: [`../../references/CONVENTIONS.md`](../../references/CONVENTIONS.md).
When this skill and CONVENTIONS disagree, **CONVENTIONS wins.** You own the
*generation method and its output shape* â€” you do **not** own `Result`/`Failure`
(skill 15), the `dioProvider`/auth interceptor (skills 11/13), or contract
pinning (skill 72). You reference them.

**Exit gate:** generated DTOs, data sources, and repositories compile under
`very_good_analysis`, `dart run build_runner build` succeeds, and repository
methods return mapped domain entities via `Result`.

## The mapping (spec â†’ house layout)

The whole skill is one idea: **map OpenAPI concepts onto the conventions** so the
output is indistinguishable from hand-written stage-11 code.

| OpenAPI concept | House artifact | Lands in |
|---|---|---|
| `components/schemas/*` | `freezed` + json DTO with `toEntity()` | `features/<f>/data/dto/` |
| `properties` + `required` | DTO fields (`required` / nullable / `@Default`) | DTO factory |
| snake_case field names | `@JsonKey(name: 'snake_case')` on camelCase field | DTO |
| `paths/*` operations (`operationId`) | data-source method (dio call, throws) | `features/<f>/data/` |
| `tags` | one data source + one repository per tag | `features/<tag>/` |
| `requestBody` schema | typed request DTO param (`dto.toJson()`) | data-source method |
| `responses.2xx.schema` | method return type â†’ DTO â†’ `toEntity()` | data source / repo |
| HTTP status / `DioException` | `Failure` via skill 15's `mapError` | repository impl |
| `securitySchemes` (bearer) | auth interceptor seam (skill 13) | `core/network/` |

Generated DTOs **carry `toEntity()`**; the domain entity it returns is *not*
generated by default (the spec describes transport, not your domain). The
generator emits an entity stub next to the DTO; you refine it, or wire `toEntity`
to a hand-written `domain/` entity. The domain never imports the DTO.

## Which generator to use

Pick by spec size and how much you want to own:

| Option | Use when | Trade-off |
|---|---|---|
| **Bundled `gen_openapi.py`** (this skill) | small/medium spec, you want house-shaped output you fully control and can diff | minimal â€” DTO stubs only; you add data sources/repos from templates |
| **`swagger_dart_code_generator`** (build_runner) | medium spec, want DTOs + Retrofit-style clients regenerated on `build_runner` | output is *its* shape; you wrap it behind house repositories returning `Result` |
| **`openapi-generator`** (`dart-dio` / `dart-dio-next`) | large/complex spec, OpenAPI 3 with rich schemas, multi-package | heaviest; generated client is a dependency you adapt, not edit |

Default to the **bundled generator** for the DTO layer when you want full control
and clean diffs, and hand-build data sources/repositories from the
[`templates/`](templates) (which mirror skill 11). Reach for
`swagger_dart_code_generator` or `openapi-generator` when the spec is large enough
that regenerating everything beats maintaining it. **In all three cases the rule
is the same: the generated client stays in `data/`, wrapped by a house repository
that returns `Result<T>` and maps exceptions to `Failure` (skill 15).** Never let
generated `dio`/JSON types leak past the repository boundary.

## Workflow

1. **Get the spec.** Save the contract at `api/openapi.json` (or `.yaml`) in the
   repo. This file is the pinned contract (skill 72) â€” commit it.
2. **Generate DTOs.** Run the bundled generator:

   ```bash
   python3 "${CLAUDE_SKILL_DIR}/scripts/gen_openapi.py" api/openapi.json \
     > lib/features/_generated/dto/api_dtos.dart
   ```

   It emits one `@freezed` DTO class per `components/schemas` entry, with
   `@JsonKey` for snake_case fields, nullability from `required`, and a
   `toEntity()` stub. Review the output, then split per-feature if you prefer.
3. **Add the codegen parts.** Run `dart run build_runner build --delete-conflicting-outputs`
   so `freezed`/`json_serializable` emit the `.freezed.dart`/`.g.dart` parts.
4. **Add data sources + repositories.** For each tag, copy
   [`templates/generated_data_source.example.dart`](templates/generated_data_source.example.dart)
   and [`templates/generated_repository.example.dart`](templates/generated_repository.example.dart),
   rename to the tag, and fill in the operations the generator listed. The
   repository wraps the data source, maps exceptions via skill 15, and exposes a
   `@riverpod` provider (skill 11's DI chain: dio â†’ data source â†’ repository).
5. **Wire auth.** Generated data sources use the shared `dioProvider`; the bearer
   token from the spec's `securitySchemes` is attached by the auth interceptor
   (skills 11/13) â€” the generator does **not** put tokens in code.
6. **Verify the gate.** `flutter analyze` clean, `build_runner` green, repository
   methods return `Result`. Then commit the spec + generated output together.

See [`templates/generated_dto.example.dart`](templates/generated_dto.example.dart)
for exactly what step 2 emits (post-review).

## Regeneration & contract pinning (ties to skill 72)

- **The spec is committed and is the contract.** Treat `api/openapi.json` as a
  pinned artifact. A backend change means a new spec â†’ regenerate â†’ review the
  diff. Skill 72 (Contract/Golden Drift) compares the live API against this
  pinned spec and fails CI when they diverge; this skill is what produces code
  *from* that pinned spec.
- **Regen is diff-driven.** Re-run the generator into the same path and inspect
  the diff. A new field, a changed type, or a removed endpoint shows up as a
  reviewable change â€” not a surprise at runtime. Keep generated files in a
  clearly-named folder (e.g. `features/_generated/` or `data/dto/`) so the diff
  is obvious.
- **Never hand-edit generated DTOs.** Edits are lost on regen. Put custom logic
  in the `toEntity()` target (the domain entity) or in the repository, both of
  which are hand-owned. If you must tweak a DTO, change the spec and regenerate.
- **Pin the generator version too.** Whichever generator you choose
  (`gen_openapi.py` here, or the pub/CLI tools), record its version so output is
  reproducible across machines and CI (skill 21).

## What stays hand-owned (do not generate)

- **`Result`/`Failure` + `mapError`** â€” skill 15. The generator's repositories
  *import* these; they never redefine them.
- **Domain entities** â€” the spec describes the wire, not your domain model.
  `toEntity()` is generated as a stub; you own the entity it targets.
- **The `dioProvider` and interceptors** â€” skills 11/13. Data sources watch the
  existing `dioProvider`; auth/logging/resilience interceptors are attached there.
- **Business logic** â€” controllers/notifiers (skill 08) consume the repository's
  `Result`; none of that is generated.
